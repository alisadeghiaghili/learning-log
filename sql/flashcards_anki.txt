<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">In what order is a SQL query logically executed?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. SELECT is evaluated last!</div>	domain::sql course::advanced_sql section::ch1_sql_execution_order
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why can't you use a SELECT alias in WHERE?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Because WHERE is evaluated before SELECT. The alias doesn't exist yet. You must repeat the expression or use a subquery/CTE.</div>	domain::sql course::advanced_sql section::ch1_sql_execution_order
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between INNER JOIN, LEFT JOIN, and FULL JOIN?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">INNER = only matching rows | LEFT = all left rows + matches | FULL = all rows from both sides. Unmatched rows get NULLs.</div>	domain::sql course::advanced_sql section::ch1_join_types
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is a CROSS JOIN and when is it used?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Cartesian product — every row from A paired with every row from B. E.g., 10 customers × 3 products = 30 rows. Used for generating all combinations like scheduling.</div>	domain::sql course::advanced_sql section::ch1_join_types
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is a SELF JOIN? Give an example.</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Joining a table to itself. E.g., finding an employee and their manager: SELECT e.name, m.name FROM employees e LEFT JOIN employees m ON e.manager_id = m.id.</div>	domain::sql course::advanced_sql section::ch1_join_types
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When is a Nested Loop Join the best choice?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When the outer input is small and there's an index on the inner table. Each outer row is looked up via index. Cost: O(N × log M).</div>	domain::sql course::advanced_sql section::ch1_join_algorithms
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">How does a Hash Join work?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">1) Build a hash table from the smaller table on the join key, 2) probe it with each row from the larger table. Cost: O(N + M). Best for two large tables with no useful index.</div>	domain::sql course::advanced_sql section::ch1_join_algorithms
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When is a Merge Join used?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When both inputs are already sorted (e.g., from an index). Both streams are traversed in parallel. Cost: O(N + M) if pre-sorted.</div>	domain::sql course::advanced_sql section::ch1_join_algorithms
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is a Correlated Subquery?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">A subquery that references the outer query — executes once per row. Usually slow unless there's an index on the filter column.</div>	domain::sql course::advanced_sql section::ch1_correlated_subquery_vs_join
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between EXISTS and IN?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">EXISTS is correlated and compact (first match suffices) and NULL-safe. IN materializes the full subquery result and has weird behavior with NULLs. EXISTS is always safer and usually faster.</div>	domain::sql course::advanced_sql section::ch1_correlated_subquery_vs_join
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why does <code>NOT IN (1, 2, NULL)</code> never return anything?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Because it's equivalent to x!=1 AND x!=2 AND x!=NULL. x!=NULL is always UNKNOWN, so the whole AND becomes UNKNOWN → row is filtered out. Use NOT EXISTS instead.</div>	domain::sql course::advanced_sql section::ch1_correlated_subquery_vs_join
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What does <code>NULL = NULL</code> evaluate to?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">UNKNOWN (neither TRUE nor FALSE). So <code>WHERE col = NULL</code> never returns any rows. You must use <code>WHERE col IS NULL</code>.</div>	domain::sql course::advanced_sql section::ch1_null_semantics
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between COUNT(col) and COUNT(*)?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">COUNT(col) ignores NULL rows. COUNT(*) counts all rows, including NULLs.</div>	domain::sql course::advanced_sql section::ch1_null_semantics
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is <code>NULL * 0</code>?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">NULL. Any arithmetic operation with NULL yields NULL. Even 0 * NULL is NULL.</div>	domain::sql course::advanced_sql section::ch1_null_semantics
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between UNION and UNION ALL?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">UNION combines results and removes duplicates (requires sort/hash). UNION ALL keeps duplicates (faster). Use UNION ALL unless you actually need dedup.</div>	domain::sql course::advanced_sql section::ch1_union
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Does SQL work on Sets or Bags?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Bag (multiset). SQL tables can have duplicate rows, but mathematical relations cannot. SELECT DISTINCT converts a bag to a set.</div>	domain::sql course::advanced_sql section::ch1_set_vs_bag
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why is SELECT * bad in production?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">1) Extra data (bandwidth, memory), 2) App breaks if schema changes, 3) Covering index won't work, 4) Query intent is unclear.</div>	domain::sql course::advanced_sql section::ch1_interview_keys
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between DELETE vs TRUNCATE vs DROP?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">DELETE = remove specific rows (with WHERE), slow, triggers fire. TRUNCATE = remove all rows, fast, triggers don't fire. DROP = remove the entire table and its schema.</div>	domain::sql course::advanced_sql section::ch1_interview_keys
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is a B-tree index and what operations does it support?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">A balanced tree structure that keeps data sorted. Supports equality lookup (O(log N)), range scans (O(log N + K)), ORDER BY, and prefix matching (LIKE 'abc%'). It's the default index in most databases.</div>	domain::sql course::advanced_sql section::ch2_btree_index_fundamentals
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why can't a B-tree index help with <code>WHERE name LIKE '%Alice'</code>?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Leading wildcard forces a full scan — the index is sorted, so you can't skip to the middle. Leading wildcards like <code>LIKE 'Alice%'</code> CAN use the index.</div>	domain::sql course::advanced_sql section::ch2_btree_index_fundamentals
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why does <code>WHERE YEAR(created_at) = 2024</code> not use an index on created_at?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">The function wraps the indexed column — the optimizer can't match it to the index. Fix: use a range <code>WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'</code>, or create a functional index.</div>	domain::sql course::advanced_sql section::ch2_btree_index_fundamentals
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is the leftmost prefix rule for composite indexes?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">A composite index (a, b, c) can be used for queries on <code>a</code> alone, <code>a+b</code>, or <code>a+b+c</code> — but NOT for <code>b</code>, <code>c</code>, or <code>b+c</code> alone. The index is sorted by <code>a</code> first, so skipping <code>a</code> breaks the sorted order.</div>	domain::sql course::advanced_sql section::ch2_composite_index_leftmost_prefix
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why put equality columns before range columns in a composite index?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Equality on a column narrows the scan within that level. A range stops the index scan — so columns after the range can't be used. Example: index (status, created_at) works for <code>WHERE status='active' AND created_at > '2024-01-01'</code> but NOT the reverse.</div>	domain::sql course::advanced_sql section::ch2_composite_index_leftmost_prefix
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is a covering index?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">An index containing all columns a query needs, so the DB never touches the table rows. Shows as "Using index" (MySQL) or "Index Only Scan" (PG). Dramatically reduces I/O.</div>	domain::sql course::advanced_sql section::ch2_composite_index_leftmost_prefix
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What's the difference between a clustered and non-clustered index?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Clustered index determines physical row order on disk — only one per table (usually the PK). Non-clustered index is a separate structure with pointers to rows. Non-clustered lookups need an extra step to fetch the full row.</div>	domain::sql course::advanced_sql section::ch2_index_types
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When would you use a partial (filtered) index?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When you mostly query a subset of rows (e.g., <code>WHERE status = 'active'</code>). The index only stores matching rows — smaller, faster to maintain. Supported in PostgreSQL and SQL Server.</div>	domain::sql course::advanced_sql section::ch2_index_types
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">In MySQL EXPLAIN, what does <code>type</code> column tell you?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">The access path, best to worst: system > const > eq_ref > ref > range > index > ALL. ALL = full table scan (usually bad on large tables with selective WHERE).</div>	domain::sql course::advanced_sql section::ch2_reading_explain_output
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What does "Using filesort" mean in MySQL EXPLAIN?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">The DB must perform an extra sort step because the ORDER BY can't be satisfied by the index. Often a red flag — add an index covering the ORDER BY columns.</div>	domain::sql course::advanced_sql section::ch2_reading_explain_output
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What does "Using temporary" mean in MySQL EXPLAIN?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">A temporary table was created during query execution — usually for GROUP BY, DISTINCT, or certain JOIN patterns. Can be expensive on large datasets.</div>	domain::sql course::advanced_sql section::ch2_reading_explain_output
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What does the query optimizer decide?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Access path (full scan vs index), join order, join algorithm (nested loop vs hash vs merge), aggregation strategy (hash vs sort), and parallelism. It's cost-based — finds the cheapest plan, not necessarily the fastest.</div>	domain::sql course::advanced_sql section::ch2_execution_plans_optimizer
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why would a query be fast in dev but slow in production?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">1) Different table stats — dev has 100 rows (full scan fine), prod has 10M. 2) Stale statistics — run ANALYZE. 3) Data distribution differs. 4) Index missing in prod. 5) Parameter sniffing — same query, different parameter values, cached plan is suboptimal.</div>	domain::sql course::advanced_sql section::ch2_execution_plans_optimizer
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">What is parameter sniffing?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">The DB caches a plan based on the first parameter values it sees. If later parameters have very different distributions, the cached plan may be suboptimal. Fix: RECOMPILE hint, pg_hint_plan, or plan_cache_mode settings.</div>	domain::sql course::advanced_sql section::ch2_execution_plans_optimizer
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When is a full table scan actually faster than using an index?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">When the query returns a large percentage of rows (>15-30%). The optimizer correctly chooses a full scan over index lookups + random I/O for each matched row. Not a bug — check EXPLAIN to confirm.</div>	domain::sql course::advanced_sql section::ch2_index_maintenance
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Why does <code>WHERE phone = 5551234</code> fail to use an index on a VARCHAR column?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">Implicit type cast — comparing VARCHAR to INT converts every row's value, preventing index use. Fix: <code>WHERE phone = '5551234'</code> (explicit string literal).</div>	domain::sql course::advanced_sql section::ch2_index_maintenance
<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">How do you find unused indexes?</div>	<div dir="ltr" style="font-family:Tahoma,Arial,sans-serif">MySQL: <code>SELECT * FROM sys.schema_unused_indexes</code>. PostgreSQL: <code>SELECT indexrelname, idx_scan FROM pg_stat_user_indexes WHERE idx_scan = 0</code>. Remove them to speed up writes.</div>	domain::sql course::advanced_sql section::ch2_index_maintenance
