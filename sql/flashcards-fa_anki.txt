IN نتیجه subquery را مادی‌سازی می‌کند؛ با NULL در لیست، NOT IN هیچ سطری برنمی‌گرداند. برای ستون‌های nullable از NOT EXISTS استفاده کن.</div>	domain::sql course::sql_fundamentals section::ch_1_correlated_subquery_در_مقابل_join

COUNT(*) تمام سطرها از جمله NULLها را می‌شمارد.</div>	domain::sql course::sql_fundamentals section::ch_1_سمانتیک_null

UNION ALL تکراری‌ها را نگه می‌دارد (سریعتر). مگر dedup واقعاً لازم باشد، UNION ALL استفاده کن.</div>	domain::sql course::sql_fundamentals section::ch_1_union

SELECT DISTINCT یک bag را به set تبدیل می‌کند.</div>	domain::sql course::sql_fundamentals section::ch_1_set_در_مقابل_bag
<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">چرا SELECT * در محیط production بد است؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">1) داده‌ی اضافی (پهنای باند، حافظه)، 2) اگر schema تغییر کند app خراب می‌شود، 3) covering index کار نمی‌کند، 4) هدف query نامشخص است.</div>	domain::sql course::sql_fundamentals section::ch_1_نکات_مصاحبه

TRUNCATE = حذف سریع همه سطرها — triggerهای row-level فعال نمی‌شوند، اما PostgreSQL triggerهای statement-level TRUNCATE را فعال می‌کند. <div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">B-tree index چیست و چه عملیاتی را پشتیبانی می‌کند؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">یک ساختار درختی متعادل که داده‌ها را مرتب نگه می‌دارد. جستجوی equality (O(log N))، range scan، ORDER BY و prefix matching (LIKE 'abc%') را پشتیبانی می‌کند. در بیشتر دیتابیس‌ها پیش‌فرض index است.</div>	domain::sql course::sql_fundamentals section::ch_2_اصول_b_tree_index

<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">Covering index چیست؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">یک index که شامل تمام ستون‌های مورد نیاز query است، پس دیتابیس هرگز سطرهای جدول را لمس نمی‌کند. در MySQL به شکل "Using index" و در PostgreSQL به شکل "Index Only Scan" نشان داده می‌شود. I/O را به شدت کاهش می‌دهد.</div>	domain::sql course::sql_fundamentals section::ch_2_composite_index_و_leftmost_prefix
<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">تفاوت clustered و non-clustered index چیست؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">Clustered index ترتیب فیزیکی سطرهای داده را کنترل می‌کند — در MySQL/InnoDB و SQL Server فقط یکی به ازای هر جدول (معمولاً PK). PostgreSQL clustered index پایدار ندارد: CLUSTER یک‌بار مرتب می‌کند اما حفظ نمی‌کند. Non-clustered index یک ساختار جداگانه با pointer است — lookup نیازمند fetch اضافی از جدول است مگر covering index باشد.</div>	domain::sql course::sql_fundamentals section::ch_2_انواع_index
<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">Partial (filtered) index چه وقتی مناسب است؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">وقتی بیشتر queryها روی زیرمجموعه‌ای از سطرها هستند (مثلاً `WHERE status = 'active'`). index فقط سطرهای مطابق را ذخیره می‌کند — کوچکتر و سریعتر برای maintenance. PostgreSQL و SQL Server پشتیبانی می‌کنند.</div>	domain::sql course::sql_fundamentals section::ch_2_انواع_index
<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">ستون `type` در MySQL EXPLAIN چه می‌گوید؟</div>	<div dir="rtl" style="text-align:right;font-family:Tahoma,Arial,sans-serif">مسیر دسترسی، از بهترین تا بدترین: system &gt; const &gt; eq_ref &gt; ref &gt; range &gt; index &gt; ALL. ALL = full table scan (معمولاً بد برای جداول بزرگ با WHERE حساس).</div>	domain::sql course::sql_fundamentals section::ch_2_خواندن_explain
